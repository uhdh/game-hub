# 3단 오목 자가학습 신경망 AI — 설계 문서

## 개요
`gomoku-stack.html`의 현재 최상위 AI는 손으로 짠 평가함수 + minimax(알파베타, "extreme" 난이도는 Web Worker 비동기 탐색)다. 카타고(KataGo)는 바둑 전용 엔진이라 이 게임에 그대로 붙일 수 없으므로, 대신 AlphaZero 방식(정책+가치 신경망 + MCTS + 반복 자가대국)으로 이 게임 전용 신경망을 처음부터 자가학습시킨다.

학습은 로컬 GPU(RTX 5080, 16GB VRAM)에서 수동으로 실행하며, 결과물(ONNX 모델)만 정적 사이트에 배포한다. 목표는 "카타고급"이 아니라 **자가학습으로 지금 minimax보다 확실히 강한 AI**를 얻는 것이며, 학습 소요 시간이나 최종 강함의 정도는 사전에 보장하지 않는다(참고할 벤치마크가 없는 커스텀 룰이므로 여러 세대를 반복 관찰하며 진행).

## 게임 룰 요약 (포팅 대상, `gomoku-stack.html` 기준)
- 헥스 보드, axial 좌표 반지름 5, 코너 6칸 제외 → 85칸 (`RAW_BOARD` 생성 로직)
- 플레이어당 돌 25개(`supply`), 칸마다 최대 3층까지 쌓기 가능
- 두 종류의 수: ① 빈 칸에 새 돌 배치(서플라이 소모) ② 인접 칸으로 자기 최상단 돌 이동 — 단, 목적지 높이가 출발지 높이 이하이고 이동 후 3층을 넘지 않아야 함(`isValidMoveTargetOf`)
- 승리조건 3가지(`checkWinOf`): 일직선 5개 연결(`checkFiveInRowOf`) / 3층 꼭대기 돌 5개(위치 무관, `checkFiveAtTop3Of`) / 인접한 3층 꼭대기 돌 3개가 삼각형(`checkTriangleTop3Of`)

## 범위
- 신규 디렉터리 `training/`(리포에 커밋되지만 배포 대상 아님)에 Python 학습 파이프라인 구현
- 산출물: ONNX 모델 파일 1개 + 이를 브라우저에서 구동하는 JS 추론/MCTS 모듈
- 기존 minimax 기반 난이도(easy~extreme)는 전혀 건드리지 않고 그대로 유지
- `gomoku-stack.html`에 새 최상위 난이도(가칭 "신경망")를 하나 추가
- Non-goals: 분산/멀티 GPU 학습, 사람 기보를 이용한 사전학습(부트스트랩), 서버 기반 추론(별도 API 서버 없음 — 항상 브라우저 내 추론)

## 아키텍처

### 1. 룰 엔진 포팅 (`training/game.py`)
`gomoku-stack.html`의 순수 로직 함수들(`isValidInitialPlacementOf`, `isValidMoveTargetOf`, `checkWinOf`, `placeStoneOf`, `moveStoneOf`, `getAllLegalMainMoves` 등)을 Python으로 1:1 재구현. JS 쪽과 동일한 키 규칙(`"{q}_{r}"`)과 동일한 인접/직선 방향 벡터(`NEIGHBOR_DIRS`, `LINE_DIRS`)를 사용해 두 구현이 항상 같은 결과를 내도록 맞춘다.

### 2. 상태 인코딩
- 85칸을 담을 수 있는 사각 격자(예: 11×11, axial q,r → 격자 인덱스 매핑)에 각 칸당 6채널로 인코딩: {바닥층-P1, 바닥층-P2, 중간층-P1, 중간층-P2, 꼭대기층-P1, 꼭대기층-P2}
- 추가 평면: 유효칸 마스크(코너 제외), 현재 차례 플레이어, 양 플레이어 서플라이 잔량(정규화된 상수 평면)
- 행동 공간: 배치 85개 + 이동(85칸 × 6방향, 이론상 최대 85+85*6=595, 비인접/비유효 조합은 항상 마스킹) — 정책 헤드는 이 전체 행동 벡터에 대해 소프트맥스 출력 후 합법수만 마스킹·재정규화

### 3. 신경망 (`training/model.py`)
- 작은 ResNet: 입력 conv → residual block 6~10개(64~128채널) → 정책 헤드(conv → FC → 행동 공간 크기) / 가치 헤드(conv → FC → tanh 스칼라)
- 보드가 작으므로(85칸) 표준 바둑용 AlphaZero 넷보다 훨씬 작은 규모로 시작하고, 학습 진행을 보며 필요시 확장

### 4. MCTS (`training/mcts.py`)
- PUCT 방식(정책망의 사전확률 P + 방문횟수 기반 탐색/활용 균형)
- 자가대국 시 루트 노드에 Dirichlet 노이즈를 섞어 탐험성 확보
- 시뮬레이션 종료 시 가치망 출력을 그대로 리프 평가로 사용(별도 롤아웃 없음)

### 5. 학습 루프 (`training/train.py`)
1. 현재 최고 모델 + MCTS로 자가대국 다수 실행 → (상태, MCTS 방문분포, 최종 승패) 튜플을 리플레이 버퍼에 저장
2. 버퍼에서 배치 샘플링 → 정책은 교차엔트로피, 가치는 MSE(+L2 정규화)로 학습
3. 신규 체크포인트 vs 현재 최고 모델로 평가 대국(예: 40~100판) 실행
4. 신규 모델 승률이 임계값(예: 55%) 이상이면 "최고 모델"로 승격, 아니면 폐기하고 다음 세대로
5. 1~4를 반복하며 학습 곡선(승률, 손실)을 관찰해 계속 진행할지 판단

### 6. Export (`training/export_onnx.py`)
최종 승격된 체크포인트를 ONNX로 변환해 `assets/models/gomoku-stack-net.onnx`(경로는 구현 시 확정)에 배치.

### 7. 웹 통합 (`gomoku-stack.html`)
- 난이도 선택 UI에 새 항목 추가(기존 easy/normal/hard/extreme 옆)
- 선택 시 onnxruntime-web으로 ONNX 세션 로드 → JS로 재구현한 PUCT MCTS(Python MCTS와 동일 알고리즘, 상태 인코딩도 동일 규칙)로 수 계산
- **폴백**: 모델 로드 실패(네트워크 오류, WASM 미지원 등)나 추론 중 예외 발생 시 기존 `computeAIMainMoveHard`(extreme 난이도 로직)로 자동 전환, 콘솔에 사유 로그만 남기고 사용자 플레이는 끊기지 않게 처리

## 검증 전략
1. **룰 포팅 정합성 검증(최우선)**: Node로 무작위 대국 N판을 두면서 매 수마다 보드 상태를 로그로 남긴 뒤, 동일한 수순을 Python 포팅 로직에 입력해 매 스텝 상태가 완전히 일치하는지 대조. 여기서 어긋나면 이후 학습 전체가 무의미하므로 학습 착수 전 반드시 통과해야 함
2. **파이프라인 스모크 테스트**: 아주 작은 네트워크 + 적은 시뮬레이션 수로 자가대국→학습 한 사이클을 먼저 돌려 에러 없이 끝까지 도는지 확인한 뒤에만 본 학습(장시간)에 투입
3. **웹 통합 후 수동 테스트**: 브라우저에서 새 난이도를 선택해 실제 대국해보며 정상 동작(합법수만 두는지, 승리조건 판정 일치, 폴백 동작)을 확인

## 리스크 / 열린 질문
- 학습에 필요한 세대 수와 시간은 사전에 알 수 없음 — 진행하며 승률 추이를 보고 계속/중단 판단
- 자가대국만으로 학습하므로 초기 세대는 매우 약할 수 있음(무작위에 가까움) — 정상적인 과정
- 브라우저 내 MCTS+ONNX 추론 성능이 기대에 못 미치면(저사양 기기 등) 시뮬레이션 횟수를 낮추는 식으로 조정 필요
