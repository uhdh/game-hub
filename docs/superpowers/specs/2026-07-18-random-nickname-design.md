# 랜덤 닉네임 자동 배정 (design)

## 배경

`gomoku-stack.html`, `rearrange.html`(피의게임 재배치), `blind-auction.html` 세 게임 모두 리더보드에
닉네임을 표시한다. 닉네임 입력창을 비운 채 제출하면 세 게임 모두 문자열 리터럴 `"익명"`을
`localStorage`와 Supabase 리더보드 테이블에 그대로 저장한다. 서로 다른 실제 사용자가 각자
비워서 제출하면 모두 같은 `"익명"` 행으로 전적이 합산되어, 예를 들어 3단 오목 상 난이도
리더보드에서 `익명`이 38승 138패로 게임 수가 가장 많은 계정처럼 보이는 문제가 발생한다.

## 목표

닉네임을 비워둔 사용자에게 기기별로 고유하게 유지되는 랜덤 닉네임을 자동 배정해서,
서로 다른 사용자의 전적이 하나의 `"익명"` 행으로 뭉치지 않게 한다.

## 범위

리더보드가 있는 3개 게임 전부: `gomoku-stack.html`, `rearrange.html`, `blind-auction.html`.
과거에 이미 `"익명"`으로 쌓인 리더보드 데이터의 소급 정리는 이번 작업 범위 밖.

## 이름 생성 로직

**단어 풀 (21개)**:
`이상민, 정근우, 박지민, 이태균, 하승진, 현성주, 윤비, 이진형, 홍진호, 서출구, 최혜선, 허성범,
김경훈, 김유현, 김남희, 강지후, 곽범, 이관희, 신승용, 최연청, 덕후`

**생성 방식**: 풀에서 무작위로 단어 하나를 고른 뒤 `#` + 무작위 정수(1~999)를 붙인다.
예: `이상민#57`, `덕후#42`.

숫자 접미사를 항상 붙이는 이유: 단어 풀이 21개뿐이라 사용자 수가 늘면 접미사 없이는
같은 이름이 자주 겹쳐 문제가 재발한다. 숫자 접미사로 사실상 유일성을 확보한다.

세 파일 모두 공유 JS 파일이 없고 각자 `<script>`에 로직을 인라인으로 갖고 있으므로(기존
프로젝트 구조), 이 이름 풀과 생성 함수를 각 파일에 동일하게 복제해 넣는다.

## 동작 방식

각 게임은 이미 자체 `localStorage` 닉네임 키를 갖고 있다:
- `gomoku-stack.html`: 난이도별 `config.nicknameKey`
- `rearrange.html`: `NICKNAME_KEY = "rearrangeNickname"`
- `blind-auction.html`: `NICKNAME_KEY = "blindAuctionNickname"`

페이지 로드 시 닉네임 입력창 초기값을 채우는 기존 로직(`localStorage.getItem(KEY) || ""`)을
다음과 같이 바꾼다:

1. 저장된 값이 있으면 → 그대로 사용 (과거에 `"익명"`으로 저장된 사용자 포함, 건드리지 않음).
2. 저장된 값이 없으면(최초 방문) → 즉시 랜덤 이름을 생성해 `localStorage`에 저장하고,
   입력창의 실제 값으로 채운다(placeholder가 아님 — 사용자가 보고 원하면 직접 수정 가능).

제출 시점의 "입력값이 비어 있으면 `'익명'`" 폴백도 동일한 랜덤 생성 함수 호출로 교체한다
(입력창이 항상 미리 채워지므로 정상 흐름에서는 거의 도달하지 않는 안전망 코드).

## 테스트 방법

자동화 테스트가 없는 순수 정적 HTML/JS 프로젝트이므로, 각 파일을 브라우저로 열어
(또는 로컬 정적 서버) 다음을 수동 확인한다:
- `localStorage` 비운 상태에서 페이지 로드 → 닉네임 입력창에 `단어#숫자` 형식 값이 자동으로
  채워지는지
- 새로고침해도 같은 이름이 유지되는지(재생성되지 않는지)
- 입력창 값을 직접 수정 후 제출하면 수정한 값이 저장/제출되는지
- `localStorage`에 기존 값(`"익명"` 포함)이 있는 상태로 로드하면 그 값이 그대로 쓰이는지

## 배포 관련 참고 (구현 세션에서 처리)

이 로컬 저장소(`모자이크퍼즐`)는 실제 배포본이 아니다. `gomoku-stack.html`은
`game-hub/apps/mosaic-puzzle/gomoku-stack.html`에서 직접 발전해 온 별도 히스토리를 가지므로
로컬 파일을 그쪽에 직접 `cp`로 덮어써서 커밋해야 하고, `rearrange.html`/`blind-auction.html`은
로컬에서 커밋 후 `git subtree pull`로 `game-hub`에 반영하는 기존 패턴을 따른다. 실제 배포
여부/시점은 구현 완료 후 사용자와 별도로 확인한다.
